### Phase 26: Test Coverage Automation & Enforcement

#### Decisions

##### Ratchet coverage baselines instead of hardcoding an aspirational 100%
gsd-orchestrator's CI gate was rewritten from a permanently-red `line-rate==100%` check to a
ratcheted `branch-rate >= 0.7314` gate (raised from a freshly re-measured 0.6913 baseline), and
autogen's gate was ratcheted to `branch-rate >= 53.5%`. Both baselines were re-measured fresh
rather than trusting the plan's stale prior figures.

**Rationale:** A permanently-red 100% target provides no actionable signal; a ratcheted gate that
can only move up (never down) gives a real, passable, non-regressing quality bar.
**Source:** .planning/phases/26-test-coverage-automation/26-01-SUMMARY.md

##### Use a detached clean worktree for cross-repo coverage verification
The final Phase 26 audit re-verified autogen's coverage from a detached worktree at the exact PR
commit rather than trusting the live (dirty) checkout.

**Rationale:** The live `autogen` checkout contained unrelated local drift and failing tests
outside the PR under audit; a clean worktree isolates the measurement.
**Source:** .planning/phases/26-test-coverage-automation/26-03-SUMMARY.md

#### Lessons

##### An operator's "continue autonomously" instruction can retroactively satisfy a blocking checkpoint
The original 26-03 plan declared a blocking human checkpoint after the coverage-audit packet was
assembled. The operator's standing instruction to continue autonomously was treated as checkpoint
approval because the packet was already complete and the remaining gate was review-only.

**Context:** This is a judgment call documented explicitly in the SUMMARY rather than silently
skipped, distinguishing "checkpoint satisfied by prior instruction" from "checkpoint ignored."
**Source:** .planning/phases/26-test-coverage-automation/26-03-SUMMARY.md

##### Locally-measured coverage figures can be superseded by authoritative remote CI
An earlier local branch-rate checkpoint (54.65%) for autogen was superseded by the authoritative
clean-branch remote CI figure (53.67%) because the local worktree contained unrelated drift not
present in the actual PR.

**Context:** Local measurements taken in a shared/dirty working tree are not trustworthy evidence
on their own; the PR's actual CI run is the source of truth.
**Source:** .planning/phases/26-test-coverage-automation/26-02-SUMMARY.md

#### Patterns

##### Real cross-platform process-executor tests instead of mocking Process
New coverage-gap-closing tests exercised `NativeProcessCommandExecutor` with real process spawns
using OS-appropriate shell one-liners (`cmd.exe /c` vs `/bin/sh -c`) rather than mocking the
`Process` class.

**When to use:** When a class's entire responsibility is orchestrating a real OS process and no
injectable abstraction exists yet — a deterministic real spawn (no network/sleep) is more honest
coverage than a mock that asserts nothing about real OS behavior.
**Source:** .planning/phases/26-test-coverage-automation/26-01-SUMMARY.md

### Phase 27: Resilience First Error Typing

#### Decisions

##### Author new schemas against the currently-live canonical host, not an assumed-future one
The `27-01` plan assumed the v1.1 schema line had already moved to the GitHub Pages registry
base. The live `cas-contracts` repo still used `schemas.coding-autopilot.dev` for v1.1, so the new
`failure-state.schema.json` was authored with that same host rather than performing a silent,
broader registry migration out of scope.

**Rationale:** Keep the deliverable aligned with the current repository contract line; defer the
host migration to its own dedicated work (later resolved in Phase 32).
**Source:** .planning/phases/27-resilience-error-typing/27-01-SUMMARY.md

#### Lessons

##### Isolate work in a dedicated worktree when the live checkout is already dirty
`27-02`'s typed-failure-state work was implemented in an isolated worktree
(`worktrees/gsd-orchestrator-phase-27`) because the live `gsd-orchestrator` checkout already had
unrelated local edits.

**Context:** This kept new failure-state work separate from the Phase 26 coverage branch and from
unrelated local modifications — though the isolation itself later became the root cause of the
work going unpushed and uncommitted for a time (see Phase 35's finding).
**Source:** .planning/phases/27-resilience-error-typing/27-02-SUMMARY.md

#### Patterns

##### Exception-type-to-failure-class classifier mapping
`FailureClassifier` maps concrete .NET exception types to a typed `FailureState` record with a
failure class and a retryable flag (e.g. `TimeoutException` -> `Transient`/retryable,
`UnauthorizedAccessException` -> `Policy`/non-retryable, unknown exception -> `Unrecoverable`).

**When to use:** Any loop/worker boundary that needs to convert unstructured exceptions into a
durable, typed, evidence-bearing failure record instead of letting them escape uncaught.
**Source:** .planning/phases/27-resilience-error-typing/27-02-SUMMARY.md

### Phase 28: Fault-Injection & Recovery Auditing

#### Lessons

##### A test-only plan can surface a real production defect
While writing fault-injection tests for checkpoint recovery, the "retry the failed state once,
then halt" contract was found broken: transient MCP exceptions were being rolled back through the
SDLC rollback cascade until the failed checkpoint settled on `Idle`. A production-code fix in
`GsdStateMachine.cs` was required to preserve the actual failed state for `ResumeAsync`.

**Context:** Fault-injection test-writing is itself a verification technique that can find bugs a
plan believed were "test-only" scope.
**Source:** .planning/phases/28-fault-injection-recovery/28-01-SUMMARY.md

#### Patterns

##### Telemetry emission that can never mask the real error
`emit_failure_telemetry()` wraps its own stderr write in `try/except Exception: return` so a
telemetry failure can never raise and mask the actual error the caller is reporting.

**When to use:** Any structured-logging/telemetry helper inserted into an existing exception
boundary — the telemetry call must be provably unable to change the caller's control flow or
exception type.
**Source:** .planning/phases/28-fault-injection-recovery/28-02-SUMMARY.md

#### Surprises

##### A dedicated worktree already contained a complete, uncommitted implementation from a prior interrupted session
When `28-02` execution began, the target worktree already held a fully-implemented
`telemetry.py`, its tests, and the corresponding wiring edits, uncommitted, apparently from an
earlier interrupted session (a stale draft SUMMARY with no commit hashes was also present).

**Impact:** The session reviewed the implementation line-by-line against the plan's behavior
spec, re-ran the full test suite, and committed/pushed it rather than re-implementing from
scratch — avoiding duplicated work, but only after independent verification, not blind trust.
**Source:** .planning/phases/28-fault-injection-recovery/28-02-SUMMARY.md

### Phase 29: Automated Peer Critic Pattern

#### Decisions

##### Adopt and independently verify pre-existing branch work rather than re-implementing
`29-01`'s deliverables (`critic.py`, `critic_cli.py`, `test_critic.py`) were found already
committed on the target branch by a prior, unpushed session. Rather than re-implementing, the
existing code was independently re-verified line-by-line against every `must_haves` truth in the
plan, the full test suite was re-run fresh (13/13 passing), and only then trusted.

**Rationale:** Blind re-implementation would waste correct prior work; blind trust without
re-verification would risk shipping unverified code as this session's own claim.
**Source:** .planning/phases/29-peer-critic/29-01-SUMMARY.md

##### Validate a false-positive-blocking threat against a real, already-reviewed PR diff
The critic was run against a real, already-open, already-reviewed autogen PR diff (#12) rather
than only synthetic fixtures, producing `0 blocking, 1 advisory` — the strongest available
falsifier for the "does the critic false-positive block good code" threat.

**Rationale:** Synthetic fixtures can only prove intended behavior; a real diff proves the critic
does not create false-positive friction on real, previously-reviewed code.
**Source:** .planning/phases/29-peer-critic/29-01-SUMMARY.md

#### Lessons

##### WSL-style paths in a worktree's `.git` backing files break Git Bash (MSYS)
A worktree's `.git` file and its `gitdir` back-pointer used `/mnt/c/...`-style paths that Git Bash
could not resolve (`fatal: not a git repository`). Rewriting both to Windows-style `C:/...`
forward-slash paths fixed `git status`/`log`/`push`/`gh pr create` without touching any tracked
repo content.

**Context:** A purely local tooling-path issue, distinct from any repo content problem — worth
recognizing quickly rather than assuming repo corruption.
**Source:** .planning/phases/29-peer-critic/29-01-SUMMARY.md

#### Patterns

##### BLOCKING vs ADVISORY severity split prevents probabilistic pattern-matching from silently blocking a PR
The critic's bare-except check is BLOCKING (deterministic), while missing-telemetry and
oversized-diff checks are ADVISORY — so a pattern match that could plausibly be a false positive
never halts a PR on its own.

**When to use:** Any automated critic/gate built on heuristic pattern matching, where some check
classes are deterministic-and-safe-to-block and others are probabilistic-and-should-only-advise.
**Source:** .planning/phases/29-peer-critic/29-01-SUMMARY.md

### Phase 30: Release Train & Branch Hygiene

#### Decisions

##### Abort a stuck interactive rebase and hard-reset to origin/main rather than resolve history conflicts
A background `git rebase` for autogen PR 10 hit a `requirements.txt` conflict and dropped into a
non-interactive-terminal-blocking `vim` commit-message prompt. Both the `autogen` and
`gsd-orchestrator` PR branches were hard-reset directly to `origin/main`, the intended fixes were
manually reapplied, committed non-interactively, and force-pushed.

**Rationale:** A background/non-interactive terminal cannot service an interactive editor prompt;
sidestepping commit-history conflicts via reset-and-reapply was faster and safer than trying to
service `vim` non-interactively.
**Source:** .planning/phases/30-release-train-and-branch-hygiene/30-02-SUMMARY.md

#### Lessons

##### `enforce_admins` relaxation alone does not bypass a required-approving-review rule
Even after temporarily relaxing `enforce_admins`, GitHub rejected an admin merge attempt on
`.github` PR #13 with "At least 1 approving review is required by reviewers with write access,"
because `required_approving_review_count = 1` is a separate protection setting.
`enforce_admins` was immediately re-enabled after the failed attempt.

**Context:** These are two independent branch-protection controls; disabling one does not disable
the other. A genuine human approval is required, not just admin rights.
**Source:** .planning/phases/30-release-train-and-branch-hygiene/30-01-SUMMARY.md

##### Branches can appear clean (`@{u}` in sync) while actually missing from origin entirely
10 of 12 branches targeted for PR creation were discovered missing from `origin` despite clean
local `@{u}` tracking-ref counts, and had to be pushed first before a PR could be opened.

**Context:** A stale local tracking ref can silently misreport a branch as "already pushed" when
it never reached the remote — a trap for any automation that trusts `@{u}` state alone.
**Source:** .planning/phases/30-release-train-and-branch-hygiene/30-03-TRAIN-REPORT.md

#### Patterns

##### Merge-train runbook + supervised driver script for large-batch cross-repo merges
A documented merge-train runbook plus a supervised driver script coordinated 25 PRs and 12
parked-branch PRs across 13 repos, with agent sessions explicitly never touching
`enforce_admins` (policy-blocked by design) and operator-side merges using the documented admin
path.

**When to use:** Any large batch of cross-repo PR merges where branch protection must stay intact
and agent sessions must not self-approve or bypass review.
**Source:** .planning/phases/30-release-train-and-branch-hygiene/30-03-TRAIN-REPORT.md

### Phase 31: Org-wide CI & Supply-Chain Hardening

#### Decisions

##### Re-resolve SHA pins live against tags actually present in target files, not a plan-authored snapshot
Across plans 31-02, 31-03, 31-04, and 31-05, the plan's pre-resolved SHA table (dated 2026-07-06)
was found stale by execution time in every case — dependabot had already bumped every target
repo's action tags forward (e.g. `checkout@v4` to `@v7`, `codeql-action@v3` to `@v4`). Every plan
re-resolved fresh SHAs via `gh api repos/<owner>/<action>/commits/<tag>` against the tags actually
present in the files, rather than applying the plan's table verbatim.

**Rationale:** Applying a stale SHA table would silently pin actions to older versions than what
was already live in the files, effectively downgrading them.
**Source:** .planning/phases/31-org-ci-supply-chain-hardening/31-02-SUMMARY.md

#### Lessons

##### A plan's "missing permissions" audit finding can go stale between authoring and execution
Multiple plans in this phase (31-02 for autogen, 31-03 for cas-contracts, 31-04 for
gsd-orchestrator) found that files the plan's audit flagged as missing `permissions:` blocks
already had the exact blocks needed by execution time — added upstream between the plan's audit
and this run. Each plan verified file-by-file before editing rather than applying the plan's
instructions verbatim, which would have produced duplicate-key no-ops.

**Context:** A time-of-audit snapshot can drift from live repo state by the time a plan executes,
especially in a fast-moving portfolio with multiple concurrent sessions and dependabot activity.
**Source:** .planning/phases/31-org-ci-supply-chain-hardening/31-04-SUMMARY.md

#### Patterns

##### Isolate per-repo remediation in a throwaway worktree branched from a freshly-fetched origin/default
Every remediation plan in this phase (31-02, 31-03, 31-04) used
`git worktree add <path> -b <branch> origin/<default>` rather than branching in the existing
checkout, because target repos had unrelated dirty working-tree state or were parked on unrelated
branches. Worktrees were removed after pushing; original checkouts were verified untouched.

**When to use:** Any remediation task touching a repo whose live checkout may carry
foreign/unrelated uncommitted work that must never be staged, committed, or discarded.
**Source:** .planning/phases/31-org-ci-supply-chain-hardening/31-03-SUMMARY.md

#### Surprises

##### Nested `.claude/worktrees/` artifacts from a prior phase's concurrent runs shadowed lint results
Running `workflow-lint.ps1` against Promptimprover, autogen, and cas-reference-product picked up
unfixed, stale copies inside nested `.claude/worktrees/` directories left over from concurrent
Phase 28 agent runs, initially making the repos appear unfixed. `git show <branch>:<file>`
confirmed the actual PR branch content was fully SHA-pinned; the nested artifacts were flagged as
deferred cleanup, not fixed in-scope.

**Impact:** A verification-only task had to develop an extra confirmation step (direct git
inspection instead of trusting a recursive filesystem lint) to avoid a false "unfixed" verdict
caused by leftover artifacts from an unrelated earlier phase.
**Source:** .planning/phases/31-org-ci-supply-chain-hardening/31-06-SUMMARY.md

### Phase 32: Contracts Registry Publishing

#### Decisions

##### Rewrite schema $id to equal its resolvable fetch URL, reversing a prior canonical-namespace decision
All 22 cas-contracts schema `$id` values were rewritten from the dead
`schemas.coding-autopilot.dev` namespace to the live GitHub Pages registry URL, explicitly
documented as a BREAKING reversal of the v1.1.1 decision to use a separate canonical namespace.

**Rationale:** `gh api .../pages` confirmed no custom domain (`cname: null`), and `curl` confirmed
the old domain no longer resolves while the Pages URL returns 200 today — identity should equal a
location that actually works.
**Source:** .planning/phases/32-contracts-registry-publishing/32-01-SUMMARY.md

##### Do not self-apply a review-gate label on your own PR
`cas-contracts` PR #18's "Classify schema compatibility" check requires a human-applied
`compatibility-reviewed` label to pass (by design, for `$id`-changing schema edits). The agent's
attempt to add that label via `gh pr edit` was denied by the environment's permission system as
self-approval of a review gate on a PR it authored; the PR was left open with the check red and
recorded as a human-action blocker instead.

**Rationale:** Applying that label is functionally a review approval, not a PR-authoring action,
and is out of scope for a "PR-only, never merge/approve" agent.
**Source:** .planning/phases/32-contracts-registry-publishing/32-01-SUMMARY.md

#### Lessons

##### A live registry-fetch smoke check is additive to, not a replacement for, an offline vendored-contract check
cas-evals already had an offline check (pinned bytes + sha256 + `$id` string, no network call).
The new `registry_check.py` live-fetch smoke check was added alongside it, not in place of it,
because the two prove different things: byte-integrity vs. actual network resolvability.

**Context:** Recognizing that two verification mechanisms answer different questions prevents
accidentally deleting a still-valuable check when adding a new one.
**Source:** .planning/phases/32-contracts-registry-publishing/32-01-SUMMARY.md

### Phase 33: Azure Infra Hardening

#### Decisions

##### Enable a stricter lint rule at "warning," not "error," when no CI enforcement gate exists yet
`use-recent-api-versions` was enabled at `warning` level (not `error`) in both cas-platform and
cloud-security-service-model, because neither repo yet has a CI gate that could safely enforce a
hard failure with a rollback path.

**Rationale:** Flipping straight to a hard-fail level with no CI gate and no rollback path risks a
first-time hard failure with no recovery mechanism.
**Source:** .planning/phases/33-azure-infra-hardening/33-01-SUMMARY.md

##### Select the oldest acceptable API version for frequently-evolving resource schemas
When multiple GA API versions satisfied the freshness linter for networking resources
(NSG + VNet), the OLDEST acceptable version (`2024-10-01`) was pinned rather than the newest, to
minimize schema-drift risk on a resource class that evolves quickly.

**Rationale:** For volatile schema categories, an older-but-still-fresh-enough version is a safer
choice than chasing the newest, since newer versions are more likely to introduce breaking schema
changes shortly after release.
**Source:** .planning/phases/33-azure-infra-hardening/33-02-SUMMARY.md

#### Lessons

##### A lint-suppression comment must be re-validated whenever the rule's global level changes, not assumed correct
The `#disable-next-line use-recent-api-versions` suppression on a preview-API resource in
`container-apps.bicep` was re-checked (not just left alone) after the rule was flipped from `off`
to `warning`, confirming it was still the only relevant suppression needed and still justified.

**Context:** A suppression comment written while a rule was off could silently become stale or
unnecessary once the rule is actually enforced; changing the rule's level is the trigger to
re-verify every existing suppression against it.
**Source:** .planning/phases/33-azure-infra-hardening/33-01-SUMMARY.md

#### Surprises

##### A coordinator-relayed signal about parallel work did not correspond to any observable repo state
A mid-execution message asserted a parallel Gemini session had already created a specific branch
in cas-platform. Investigation (`git reflog`, `git worktree list`, `git ls-remote`, `gh pr list`)
found zero evidence of any such branch, worktree, or PR. The session proceeded as sole author and
recorded the investigation rather than silently ignoring or blindly trusting the signal.

**Impact:** Coordinator-relayed cross-session signals are not always accurate and should be
independently verified against live git/GitHub state before acting on them.
**Source:** .planning/phases/33-azure-infra-hardening/33-01-SUMMARY.md

### Phase 34: Workspace Guardrails & Drift Prevention

#### Decisions

##### An explicit executor instruction can supersede a plan's own stated scope deferral
`34-01-PLAN.md` explicitly deferred GLOBAL_AGENTS.md changes to `34-02`, but a direct executor
instruction during the session called for committing the Azure hard-lock section and adding a new
Workspace Hygiene Locks section within `34-01` itself. The instruction was followed and the
divergence from the plan file's stated scope was documented explicitly in the SUMMARY.

**Rationale:** A live, direct instruction takes precedence over a plan file's pre-written scope
note, but the divergence must be recorded, not silently absorbed.
**Source:** .planning/phases/34-workspace-guardrails/34-01-SUMMARY.md

#### Lessons

##### PowerShell 5.1's default OEM-codepage console encoding corrupts non-ASCII git output
PS 5.1 defaults `[Console]::OutputEncoding` to the legacy OEM codepage (`ibm850` on this machine),
which mis-decodes UTF-8 bytes git writes for non-ASCII paths/branches/config values — concretely,
this workstation's own non-ASCII user profile directory (`C:\Users\KimHarjamäki`) triggered a
false `worktree-missing` finding for a repo's own primary worktree. Fixed by forcing
`[Console]::OutputEncoding` and `$OutputEncoding` to UTF-8 at script start, before any git call.

**Context:** This is a general-purpose, real-condition fix (not a test-only workaround) — the
actual workstation path contains an umlaut, so any PS 5.1 script that shells out to git without
this fix is at risk of the same class of false finding.
**Source:** .planning/phases/34-workspace-guardrails/34-01-SUMMARY.md

##### PowerShell 5.1 pipeline unrolling can turn a 1-element array return into a scalar with a null `.Count`
Under `Set-StrictMode` + `$ErrorActionPreference = 'Continue'`, returning a 1-element array from a
function unrolled it to a scalar, whose `.Count` then resolved to `$null` rather than `1`. Fixed
with the comma operator (`return ,$violations.ToArray()`).

**Context:** This root-caused 3 of 5 initially-failing Pester red-fixture tests in the
commit-integrity check; the fix pattern (comma-operator array-return guard) is a general PS 5.1
hazard, not specific to this one script.
**Source:** .planning/phases/34-workspace-guardrails/34-02-SUMMARY.md

#### Patterns

##### `-Json` mode must suppress all `Write-Host` output to keep stdout machine-parseable
When a script is invoked as a nested `powershell.exe -File` child process (as CI/Pester do),
`Write-Host` output folds into the same captured stdout stream as JSON output, corrupting it. All
`Write-Host` calls are suppressed when `-Json` is set, making JSON the sole stdout content in that
mode.

**When to use:** Any PowerShell script with both a human-readable mode and a `-Json` machine mode
that may be invoked as a nested child process.
**Source:** .planning/phases/34-workspace-guardrails/34-01-SUMMARY.md

### Phase 35: Verification & Milestone Audit

#### Decisions

##### An audit must be backed by live command evidence, not SUMMARY.md narration
Every verdict in the Phase 35 verification report is backed by a command run in that session
(`gh pr view`, `gh run list`, `gh api`, `curl`, `git log --all`, `dotnet build`, `Invoke-Pester`,
`workspace-health.ps1`) against live GitHub/local state, explicitly not trusting SUMMARY.md claims
at face value.

**Rationale:** Two findings in this audit (autogen `main` CI red; gsd-orchestrator Phase 27-02
uncommitted) were not flagged in any SUMMARY.md and were only surfaced by spot-verifying live
state — proving that SUMMARY-only verification would have missed real gaps.
**Source:** .planning/phases/35-v1.4-verification-and-audit/35-VERIFICATION.md

#### Lessons

##### A SUMMARY.md's claim of "committed, tested, passing" is not durable evidence on its own
`27-02-SUMMARY.md` narrated a complete, committed, 238/238-passing implementation. Live
verification found the entire deliverable existed only as uncommitted working-tree changes in an
isolated worktree, with no commit, no push, no PR, and no origin tracking ref — roughly half of
REQ-1.4.2's evidence had zero durable trail despite the SUMMARY's narration.

**Context:** This is recorded as the single most severe finding in the audit; it was remediated
within hours of discovery (committed, pushed, PR opened), but the gap itself is the lesson: a
SUMMARY is a claim, not proof, until independently checked against git/GitHub state.
**Source:** .planning/phases/35-v1.4-verification-and-audit/35-VERIFICATION.md

##### A CI job intended to be "required and green" can be neither, silently
Root `master` branch had no GitHub branch protection at all (`404 Branch not protected`), so
neither the `pester` job nor the `commit-integrity` job (also explicitly `continue-on-error: true`
by design) was a GitHub-enforced required check, even though REQ-1.4.13 was worded as if it were.

**Context:** "Required" in a requirement's prose does not make a check required in GitHub unless
branch protection is actually configured to say so — these are two independent facts that must
both be verified.
**Source:** .planning/phases/35-v1.4-verification-and-audit/35-VERIFICATION.md

#### Surprises

##### Live-state spot-verification caught two regressions that no SUMMARY had flagged
Two live-reproducible failures (autogen `main` CI red since 2026-07-06 on a pip resolver
conflict; the gsd-orchestrator Phase 27-02 uncommitted deliverable) were found only by directly
re-running commands against live state in this audit session, not by reading any prior SUMMARY.md.

**Impact:** Confirms the audit's own stated method note is necessary practice, not
over-engineering — a milestone audit that only reads SUMMARYs would have certified a milestone
with a genuinely broken CI and a stranded, at-risk-of-loss implementation.
**Source:** .planning/phases/35-v1.4-verification-and-audit/35-VERIFICATION.md

### Phase 36: Portfolio Documentation Refresh

#### Decisions

##### Verify every factual claim against the live tree before writing it, even when a plan brief asserts it
Task briefs for this phase asserted specific facts (e.g. "ci-autopilot has a 90% coverage gate")
that live-tree verification (`grep` across `ci.yml` and the full workflow tree) found to be false.
Documentation was written to match the verified live state, not the brief's assertion.

**Rationale:** A documentation-refresh task's entire purpose is accuracy; propagating an unverified
brief-level claim into a README/wiki would create a new, durable false claim.
**Source:** .planning/phases/36-documentation-refresh/36-task2-SUMMARY.md

#### Lessons

##### A plan's "before" description of a PR can be mistaken for the current state of `main`
A first README draft stated `main`'s CI ran a "permanently-red line-rate" enforce step, language
borrowed from the plan's `<repo_facts>` describing a PR's before-state relative to itself. Direct
`grep` of `origin/main`'s `ci.yml` showed no such step exists on `main` at all — only
collect-and-upload. The section was rewritten before the first commit to describe `main`
accurately and frame the open PR as adding the first enforcement step, not replacing one.

**Context:** Plan-supplied "facts" describing a PR's diff can be ambiguous about which branch a
given state applies to; the live default branch must be checked directly before writing a claim
about it.
**Source:** .planning/phases/36-documentation-refresh/36-task1-SUMMARY.md

##### Work described elsewhere as "delivered" can exist only on an unpushed local branch with no PR
While documenting gsd-orchestrator's typed-failure retry/halt design (Phase 28-01), direct
verification (`git log --all`, `git branch -a`) found no PR exists for that work at all — it lived
only on a local, unpushed branch in the operator's own workstation checkout. The documentation was
written to state this explicitly rather than phrasing it as "in progress (PR #N)," which would
have implied a PR that did not exist.

**Context:** The natural phrasing pattern used elsewhere in the same task ("in progress (PR #N)")
would have been factually wrong here; verifying before applying a template phrase mattered.
**Source:** .planning/phases/36-documentation-refresh/36-task1-SUMMARY.md

#### Patterns

##### One detached worktree per repo, branched from a freshly-fetched origin/default, to leave dirty checkouts untouched
Both documentation-refresh tasks in this phase created one detached worktree per target repo,
branched `docs/phase-36-refresh` from `origin/<default>`, so the existing checkouts under
`portfolio/<repo>` (several with pre-existing uncommitted local changes or parked on unrelated
branches) were never touched. `git status --short` was verified unchanged on each original
checkout before and after.

checkouts may carry unrelated in-progress work that must not be disturbed.

## Cross-Phase Patterns

### Every failure class found in execution became an automated check the same week
Across the milestone, discovered failure classes (encoding hazard, stale tracking refs,
untruthful commits, WSL worktrees, unpushed work) were each turned into an automated detection
check in the same week they were found, rather than being fixed once and left to recur silently.

**Source:** .planning/milestones/v1.4-ROADMAP.md

### Parallel AI sessions strand uncommitted/unpushed work
Stated as a recurring lesson across the milestone: three separate incidents of parallel AI
sessions stranding uncommitted or unpushed work were caught and rescued (Phase 26's cherry-picked
stranded checkpoint-corruption commit, Phase 30's stranded-commit rescues, and Phase 35's rescue of
the uncommitted Phase 27-02 FailureState work) — detection of this failure class is now automated
via workspace-health.ps1's `unpushed`/`dirty` checks (Phase 34).
