Implement `function Sync-ReadOnlyWorktree($SourceRepo, $WorktreePath, $Ref)`: run `git -C $SourceRepo fetch origin main` best-effort (redirect stderr, do not fail the whole run if offline, just proceed with whatever ref state already exists locally and note it in the suite's output). If $WorktreePath does not exist: `git -C $SourceRepo worktree add --detach $WorktreePath $Ref`. If it already exists: `git -C $WorktreePath fetch $SourceRepo $Ref -q` then `git -C $WorktreePath checkout --detach FETCH_HEAD -q` (re-resolves $Ref fresh each run without requiring $Ref to already be a remote-tracking branch inside the worktree's own remote config). This function is the single point that ever touches $SourceRepo, and the only mutation is fetch (read-only) plus worktree registration metadata -- it never checks out, resets, or commits against $SourceRepo's own working tree or its currently-checked-out branch.

After running the suites selected by -OnlySuites, build a summary object: schemaVersion "1.0.0", runDate (yyyy-MM-dd), startedAt/finishedAt (ISO 8601 UTC), overallStatus ("passed" only if every run suite is "passed"), and a suites array with each suite's id, owningRepo, status, durationSeconds, commitSha. For any suite with status not "passed", add a failureExcerpt field: the last 40 lines of that suite's captured output, with every literal occurrence of $Root (e.g. C:\PersonalRepo) replaced with the literal string "<repo-root>" before storing, so no local filesystem path ever leaves the machine (this same redacted excerpt is what later gets handed to the issue filer in Task 2).

Write the full summary as pretty-printed JSON to Join-Path $EvidenceRoot "$runDate.json" (create $EvidenceRoot if missing). Write each suite's full untruncated captured output to Join-Path $LogRoot "$runDate\$suiteId.log" (create the dated subdirectory if missing) -- this directory lives under scratch\, which is already covered by the repo's blanket scratch/ gitignore rule (.gitignore line 48), so no new gitignore entry is needed.

Unless -NoCommit is set, commit the evidence JSON via a dedicated evidence worktree so the primary C:\PersonalRepo working tree (which may have unrelated uncommitted work in progress, as it does right now) is never touched: sync a worktree at $Root\worktrees\cas-workstation-pilot-cadence the same way (create-if-missing-else-fetch-and-recheckout), but target origin/master instead of origin/main. Inside that worktree, create/switch to local branch evidence/pilot-cadence-$runDate off freshly fetched origin/master, copy the evidence JSON into that worktree's evidence\pilot-cadence\$runDate.json, git add, git commit -m "docs(pilot-cadence): evidence for $runDate ($overallStatus)", git push -u origin evidence/pilot-cadence-$runDate. Then check `gh pr list --repo OgeonX-Ai/cas-workstation --head evidence/pilot-cadence-$runDate --json number`; if empty, `gh pr create --repo OgeonX-Ai/cas-workstation --base master --head evidence/pilot-cadence-$runDate --title "docs(pilot-cadence): evidence for $runDate" --body "Automated pilot-cadence evidence, overall status: $overallStatus. See evidence/pilot-cadence/$runDate.json."`; if a PR already exists for that branch (a same-day re-run), skip creation and report the existing PR URL instead. This is the concrete implementation of "root repo = PR flow now" for this phase's automation: the scheduled task never pushes to master and never touches the primary working tree's currently checked-out branch.

Print a one-line summary per suite plus the overall status and evidence file path to stdout. Exit 0 if overallStatus is "passed", else 1.

Create evidence/pilot-cadence/README.md documenting: the schema fields above; the storage split decision (dated summary JSON is git-committed here, mirroring the evidence/compliance/snapshots/ precedent; full raw suite output is written to scratch/pilot-cadence-logs/{date}/, which is gitignored, to avoid repo bloat from verbose dotnet test/pytest -v output); and the three suite ids with their owning repos.

  <name>Task 2: Dedupe-guarded regression issue filer, wired into the runner</name>
  <files>scripts/file-pilot-regression-issue.ps1, scripts/run-pilot-cadence.ps1</files>
  <action>
Create scripts/file-pilot-regression-issue.ps1, ASCII-only, CmdletBinding with params: -Repo (owning repo slug, e.g. Coding-Autopilot-System/gsd-orchestrator), -SuiteId, -FailureExcerpt, -RunDate. Compute a stable title with no date component: "pilot-cadence: $SuiteId regression" -- so repeat searches always match the same string regardless of which day it first fired.

Dedupe check: `gh issue list --repo $Repo --state open --search "in:title \"$Title\"" --json number,title,url`, parse the JSON, and filter client-side to entries where the title equals $Title exactly (gh's search endpoint does substring/token matching, not exact match, so client-side filtering is required to avoid false-positive dedupes against unrelated issues that happen to share words). If a match exists: `gh issue comment <number> --repo $Repo --body <text>` with a short note that the regression recurred on $RunDate plus the already-redacted $FailureExcerpt, print "deduped: commented on #<number>", and return the existing issue's URL.

If no match exists: `gh issue create --repo $Repo --title $Title --body <text>` with a body containing the suite id, $RunDate, and the redacted $FailureExcerpt in a fenced markdown block. Attempt `--label pilot-cadence,regression`; if gh errors because either label does not exist in the target repo, retry the gh issue create call once without --label rather than failing the whole run over a missing label. Print "filed: <url>" and return the new issue's URL.

The script must never throw on a gh call failing for a reason other than the label case above (e.g. transient network error) -- catch and print a warning, since a failed issue-file must not be allowed to mask the fact that a regression was already correctly detected and already lives in the evidence JSON and in the redacted log; the evidence JSON is the source of truth regardless of whether the issue-filing side-effect succeeds.

Wire this into scripts/run-pilot-cadence.ps1 (from Task 1): unless -NoIssueFile is set, for every suite in the summary with status not "passed", call scripts\file-pilot-regression-issue.ps1 with -Repo $suite.OwningRepo -SuiteId $suite.Id -FailureExcerpt $suite.failureExcerpt -RunDate $runDate, capture the returned issue URL, and add it to a new issuesFiled array in the evidence summary object (each entry: suiteId, issueUrl, deduped bool) before the evidence JSON is written to disk -- so the evidence file itself records exactly what issue-filing action was taken for that run.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Scheduled/unattended runner -> portfolio sub-repos (gsd-orchestrator, autogen) | The runner reads and test-executes code from origin/main of repos it does not own the review process for; must never write back |
| Scheduled/unattended runner -> root repo git history | The runner commits evidence and opens PRs against OgeonX-Ai/cas-workstation unattended, without a human reviewing the commit before it is pushed |
| Scheduled/unattended runner -> GitHub issue trackers | A false-positive or repeated regression could spam three different repos' issue trackers with unattended, unreviewed content |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-40-01 | Tampering | Sync-ReadOnlyWorktree against portfolio/gsd-orchestrator and portfolio/autogen | mitigate | The function only ever runs git fetch (read-only) against the source repo plus git worktree add / worktree-scoped checkout --detach; it never runs checkout, reset, merge, commit, or push against the source repo's own working tree or its currently-checked-out branch. |
| T-40-02 | Elevation of Privilege | dotnet test / pytest execution inside the fault-injection worktrees | accept | Test execution is read-only with respect to source under test; worktrees live under the gitignored worktrees/ root so build artifacts never risk being staged into any repo's history. Local single-developer workstation, not shared/multi-tenant. |
| T-40-03 | Denial of Service | Repeated red runs spamming GitHub issue trackers across three repos | mitigate | scripts/file-pilot-regression-issue.ps1 performs an exact-title dedupe search (gh issue list --state open) before creating; a match produces a comment instead of a new issue. One issue max per suite per open-regression window. |
| T-40-04 | Repudiation | Evidence loss or silent tampering with pass/fail history | mitigate | Dated summary JSON is git-committed via PR (immutable history once merged); full raw logs retained locally in scratch/pilot-cadence-logs/ for debugging without bloating the tracked repo. |
| T-40-05 | Information Disclosure | Local filesystem paths or excessive internal detail leaking into public/org-visible GitHub issue bodies | mitigate | $Root (e.g. C:\PersonalRepo) is string-replaced with "<repo-root>" in the failure excerpt before it is ever written to the evidence JSON or handed to the issue filer; only the last 40 lines of output are ever surfaced. |
| T-40-06 | Tampering | Root-repo evidence commits bypassing the current PR-flow policy | mitigate | The runner never checks out or commits against the primary C:\PersonalRepo working tree (also never guaranteed clean); it uses a dedicated worktrees/cas-workstation-pilot-cadence worktree, a dedicated evidence/pilot-cadence-{date} branch, and opens a PR rather than pushing to master. |

No package-manager installs (npm/pip/cargo) are introduced by this plan -- both scripts call only already-installed CLIs (git, gh, dotnet, pytest via the existing autogen venv, powershell). The Package Legitimacy Gate does not apply.
