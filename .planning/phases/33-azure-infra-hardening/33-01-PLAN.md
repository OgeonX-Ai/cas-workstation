
Use `"warning"` rather than `"error"` for this rule specifically: `az bicep lint` and
`az bicep build` do not fail the process on warnings (only on errors), and this repo has
no CI gate yet to enforce a hard failure — `"warning"` gives visible, actionable signal
without introducing a first-time hard-fail surface with no rollback path. Do not change
any other rule in the file. Do not rename the file or add a second `.bicepconfig.json` —
Bicep CLI resolves `bicepconfig.json` (no leading dot) here today; introducing a
dotted-name duplicate would create ambiguous/conflicting config resolution.
</action>

<threat_model>
Main risk: enabling `use-recent-api-versions` could, in a hard-fail (`"error"`) config or
future CI gate, break every existing environment's deployability the moment a resource
provider's API version ages past 730 days, even though nothing about the resource's actual
behavior changed. Mitigation: this plan uses `"warning"`, not `"error"`, specifically so a
future routine "your API version just turned 731 days old" event is visible but never a
hard build/deploy blocker without a human explicitly re-tightening the rule level in a
follow-up change. Secondary risk: assuming the `#disable-next-line` suppression in
container-apps.bicep is still correct without re-checking it once the rule is live — Task 3
explicitly re-validates this rather than assuming Task 1/2's local scratch-test results
(captured during planning, dated 2026-07-06) hold unchanged at execution time, since Azure
API version freshness windows are date-relative and the plan may execute on a later date.
Tertiary risk: silently reintroducing the P1 regression (hardcoded
`publicNetworkAccess: 'Enabled'`) by branching from stale local `main` instead of
`origin/main` — Task 1 is a required verification gate specifically to catch this before
any P2 work proceeds, and its acceptance criteria require an empty diff against
`origin/main` on the P1-related files if no fix was needed.
</threat_model>
