<objective>
Build the org-wide workflow-lint falsifier for REQ-1.4.10: a PowerShell 5.1-safe, ASCII-only
script that scans GitHub Actions workflow YAML for three compliance signals — third-party
`uses:` steps pinned to a 40-character commit SHA (not a mutable tag), a `permissions:` block
present (repo-level or per-job), and `timeout-minutes:` present on every job — and exits
non-zero with a findings report when any workflow fails.

Purpose: This script is the acceptance test for every other plan in Phase 31. Wave 2 plans
remediate the 13 portfolio repos; wave 3 re-runs this script against every remediated repo to
prove REQ-1.4.10 is satisfied org-wide. It must exist and be proven correct (via its own test)
before remediation starts, so remediation plans have a concrete, runnable check rather than
a hand-inspection.

Output: `scripts/workflow-lint.ps1` (the linter) and `tests/Workflow.Lint.Tests.ps1` (falsifier
proving both the pass path and the fail path using generated fixture YAML — no dependency on
portfolio repo state, so it runs standalone in root CI).
</objective>

<task type="auto" tdd="true">
  <name>Task 1: Write scripts/workflow-lint.ps1</name>
  <files>scripts/workflow-lint.ps1</files>
  <behavior>
    - Given a directory tree containing `**/.github/workflows/*.yml` (and `.yaml`), for each
      workflow file the script must detect and report three independent violation classes:
      1. `unpinned-action`: any `uses: <owner>/<repo>...@<ref>` step where `<ref>` is NOT a
         40-character lowercase hex commit SHA (tags like `@v4`, `@v4.2.2`, branch names, and
         local/reusable workflow refs like `uses: ./`  or `uses: docker://...` must be
         evaluated correctly - local relative actions (`uses: ./path`) and `docker://` refs are
         exempt from SHA-pinning, they are not third-party marketplace actions).
      2. `missing-permissions`: a workflow file with no `permissions:` key at the top level AND
         no `permissions:` key nested under any job - GitHub Actions treats a workflow with
         neither as running with default (broad) token permissions.
      3. `missing-timeout`: any job block (top-level key under `jobs:`) that has no
         `timeout-minutes:` key directly under it.
    - Test 1 (pass fixture): a workflow file with `actions/checkout@34e114876b0b11c390a56381ad16ebd13914f8d5 # v4`,
      a top-level `permissions: {contents: read}`, and every job carrying `timeout-minutes: 10`
      must produce zero findings and the script must exit 0.
    - Test 2 (fail fixture): a workflow file with `actions/checkout@v4` (tag, not SHA), no
      `permissions:` anywhere, and a job with no `timeout-minutes:` must produce three findings
      (one per class) and the script must exit 1, with the findings printed to the console.
    - Test 3 (exemption edge case): a workflow file using `uses: ./.github/actions/local-thing`
      and `uses: docker://alpine:3.19` must NOT be flagged as `unpinned-action` (these are not
      SHA-pinnable third-party marketplace refs).
  </behavior>
  <action>
    Create `scripts/workflow-lint.ps1` as a `[CmdletBinding()]` script matching the
    workspace-health.ps1 shape (ASCII-only source, PowerShell 5.1-compatible syntax - no `??`,
    no `?.`, no ternary operator, use `New-Object System.Collections.Generic.List[object]` not
    generic collection literals). Parameters: `-Path` (string, default `.`, root directory to
    scan - callers pass a single repo path or the whole `portfolio` tree), `-Json` (switch, emit
    JSON instead of a table). Algorithm: recursively find `*.yml`/`*.yaml` under
    `<Path>/**/.github/workflows/` via `Get-ChildItem -Recurse -Include *.yml,*.yaml -Path
    (Join-Path $Path '.github/workflows')` per discovered repo directory (a repo directory is
    any directory directly containing a `.github/workflows` subfolder - walk `-Path` looking for
    `.github/workflows` folders at any depth so a single invocation against the whole
    `portfolio/` tree finds all 13 repos' workflow folders in one pass). For each workflow file:
    read raw text lines for the `uses:` regex scan (do not fully YAML-parse for this check -
    line-based regex is sufficient and avoids a YAML module dependency); use
    `ConvertFrom-Yaml`-free line scanning: match `^\s*(?:-\s*)?uses:\s*(\S+)` per line, skip
    matches starting with `./` or `docker://`, then test whether the value's suffix after the
    last `@` matches regex `^[0-9a-f]{40}$` - if not, record `unpinned-action` with the file
    path, line number, and the raw ref. For `missing-permissions`: check whether the raw text
    contains a top-level `^permissions:` line OR at least one `^\s{2,}permissions:` line nested
    under a job - if neither pattern matches anywhere in the file, record one
    `missing-permissions` finding for the file. For `missing-timeout`: parse job names as lines
    matching `^  ([A-Za-z0-9_-]+):\s*$` directly under a `^jobs:` line, then for each job block
    (from that job's line to the next sibling job or EOF) check for a `timeout-minutes:` line
    within the block - if absent, record `missing-timeout` with the file path and job name.
    Accumulate all findings into a `System.Collections.Generic.List[object]` of
    `[pscustomobject]@{ File; Check; Detail }`. Emit via `Format-Table` (default) or
    `ConvertTo-Json -Depth 4` (`-Json`). Exit 1 if any findings exist, else exit 0 and print
    "workflow-lint: clean." Use `$ErrorActionPreference = 'Continue'` at the top (per
    workspace-health.ps1's documented rationale: must survive malformed files, not die on them)
    and wrap each file's per-file scan in try/catch so one bad file does not abort the whole
    sweep - a caught parse error becomes its own `parse-error` finding for that file.
  </action>
  <verify>
    <automated>pwsh -NoProfile -Command "& { $ErrorActionPreference='Stop'; . { } }; pwsh -File scripts/workflow-lint.ps1 -Path . -Json | Out-Null; exit 0"</automated>
  </verify>
  <done>scripts/workflow-lint.ps1 exists, is ASCII-only, runs under PowerShell 5.1 syntax rules, and correctly classifies unpinned actions, missing permissions, and missing timeout-minutes when pointed at fixture workflow files.</done>
</task>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-31-01 | Denial of Service | workflow-lint.ps1 per-file scan | mitigate | Wrap each file's scan in try/catch producing a `parse-error` finding instead of throwing, so one malformed YAML file cannot abort the whole sweep or CI run |

| T-31-03 | Information Disclosure | Findings output (`-Json`) | accept | Findings are file paths and workflow content already public in the repo; no secrets are read or echoed |
</threat_model>
