<objective>
Give `gsd-orchestrator`'s core execution loop (`LoopCoordinator.RunAsync`) explicit typed failure states for unhandled exceptions, satisfying REQ-1.4.2 ("must implement explicit typed failure states for all unhandled exceptions, verified by a fault-injection E2E test" — the E2E itself is Phase 28; this plan delivers the types + wiring + unit tests that the Phase 28 E2E will exercise).

Purpose: `LoopCoordinator.RunAsync` (src/GsdOrchestrator/Loop/LoopCoordinator.cs) currently has no top-level exception handling at all — an exception from `worker.ExecuteAsync`, `verifier.VerifyAsync`, or any `store` call propagates completely unbounded, past `ITerminalOutcomePublisher`, with no classification, no evidence record, no structured log. Separately, `GsdStateMachine.ExecuteLoopAsync`'s `catch (Exception ex)` (Workflows/GsdStateMachine.cs:211-226) already classifies failures by testing `ex.Message.Contains("budget", ...)`, an unreliable string-matching heuristic that this plan replaces with type-based classification reusing the existing (currently unused) `FailureClass` enum in GoalModels.cs.
Output: A new `FailureState` record + `FailureClassifier` static class in `Loop/FailureState.cs` mapping exception `Type` -> `FailureClass` deterministically, wired into `LoopCoordinator.RunAsync` via a top-level try/catch that logs structured JSON, records an evidence entry, and republishes a `TerminalLoopOutcome` with a failed status instead of letting the exception escape uncaught.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MafProcessLoopWorker subprocess -> LoopCoordinator | Untrusted subprocess stdout/stderr/exit-code crosses into the .NET process; malformed or hostile output must not crash the host loop |
| LoopCoordinator -> structured log sink | FailureState.Message/CauseChain must not leak secrets (env vars, tokens) into logs |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-27-04 | Denial of Service | LoopCoordinator.RunAsync | mitigate | Any unhandled exception from worker/verifier/store is now caught and converted to a terminal FailureState instead of crashing the coordinating process — an unbounded exception can no longer take down the whole watch-mode loop (Program.cs's outer catch already retries the next interval; this plan ensures the inner loop fails cleanly instead of relying on that outer safety net alone) |
| T-27-05 | Information Disclosure | FailureState.Message / CauseChain in structured logs | accept | Existing exception messages already flow into `_logger.LogError(ex, ...)` today (GsdStateMachine.cs) with no additional redaction layer in this codebase; this plan does not widen that exposure (same messages, now also classified and recorded as evidence) — full secret-redaction middleware is out of scope for Phase 27 and not required by REQ-1.4.2 |
| T-27-06 | Tampering | MafProcessLoopWorker exit code / stdout | accept | Already handled upstream (non-zero exit code throws InvalidOperationException today); this plan only changes what happens to that exception once thrown, not the subprocess trust boundary itself |
</threat_model>
