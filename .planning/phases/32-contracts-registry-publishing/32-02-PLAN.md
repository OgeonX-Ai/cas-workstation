From portfolio/cas-evals/src/cas_evals/contracts.py (current, to be edited):
```python
VENDOR_DIR = Path(__file__).parents[2] / "vendor" / "cas-contracts" / "v0.1.0"
PROVENANCE_PATH = VENDOR_DIR / "provenance.json"

def verify_vendored_contract() -> dict[str, Any]:
    provenance = _load_json(PROVENANCE_PATH)
    for filename, expected in provenance["schemas"].items():
        path = VENDOR_DIR / filename
        digest = hashlib.sha256(path.read_bytes()).hexdigest()
        if digest != expected["sha256"]:
            raise ContractValidationError(f"vendored schema digest mismatch: {filename}")
    common = _load_json(VENDOR_DIR / "common.schema.json")
    evaluation = _load_json(VENDOR_DIR / "evaluation-result.schema.json")
    if common.get("$id") != "https://schemas.coding-autopilot.dev/v0.1/common.schema.json":
        raise ContractValidationError("unexpected common schema identity")
    if evaluation.get("$id") != "https://schemas.coding-autopilot.dev/v0.1/evaluation-result.schema.json":
        raise ContractValidationError("unexpected evaluation schema identity")
```
`verify_vendored_contract()` is called by `validate_evaluation_result()` (used in the evaluator pipeline) and directly by `tests/test_contracts.py::test_vendored_contract_provenance_is_valid`. Changing the two hardcoded `$id` string literals is sufficient — no signature changes.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI runner to public GitHub Pages registry | `registry_check.py` performs live outbound HTTP GETs from CI to an unauthenticated public endpoint; a compromised or spoofed DNS/CDN response could return a crafted 200 with unexpected content |
| vendored schema bytes to offline validator | `verify_vendored_contract()` trusts the sha256 recorded in `provenance.json` matches what a human reviewer approved in the PR diff; a bad actor with repo write access could update both the vendored bytes and the provenance hash together |
| CI job network dependency | `registry-smoke` job depends on the GitHub Actions runner having egress to github.io; a runner network outage would fail the job even though the registry itself is healthy |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-32-05 | Spoofing | registry_check.py HTTP responses | accept | GitHub Pages serves over HTTPS with a valid cert chain; urllib validates TLS by default; no custom SSL context is introduced, so standard cert validation is the control |
| T-32-06 | Tampering | vendor/cas-contracts/v0.1.0 re-vendor | mitigate | provenance.json sha256 is recomputed and committed alongside the file change in the same PR diff, so reviewers can see byte-for-byte what changed; verify_vendored_contract() still fails closed on any future mismatch |
| T-32-07 | Denial of Service | registry-smoke job depends on runner egress | accept | Job is isolated from the offline unit-test matrix (separate job, not a blocking step in verify); a transient network failure here does not block merges of unrelated cas-evals changes |
| T-32-08 | Information Disclosure | registry_check.py response body handling | mitigate | Follows reference_product.py's MAX_RESPONSE_BYTES cap; smoke check only inspects status code, never logs or persists response bodies |
| T-32-SC | Tampering | pip install -e . in CI | accept | No new third-party dependency added (registry_check.py is stdlib-only urllib); existing cas-evals dependency surface (dependencies = [] in pyproject.toml) is unchanged |
</threat_model>
