<objective>
Build the standalone `critic` role for REQ-1.4.5: a reusable Python module in
`maf_starter/` that statically checks a unified diff for Resilience First
violations (typed failures, try/except with structured telemetry, no bare
`except Exception`/`except:`, file size limits per the Immutable Coding
Standards SRP rule), plus a local CLI entrypoint that runs it against a
diff/PR and exits non-zero on blocking findings.

Purpose: This is the mechanical engine behind the "Peer Critic Pattern"
already named in `engineering-os/OPERATING-CONTRACT.md` (Delegation section:
"For high-risk implementations, orchestrators should spawn a concurrent,
isolated `critic` agent to perform static analysis and security review on the
`worker` agent's output prior to integration"). Today that line is aspirational
prose with no executable behind it. This plan makes it real and runnable
without requiring Azure, without requiring a paid API key (local Ollama via
`OLLAMA_BASE_URL`/`OLLAMA_MODEL` per `engineering-os/router/ollama-policy.json`
remains the project's local-model path; the deterministic pattern-scan pass
built here never needs a model at all and works fully standalone).

The critic is intentionally NOT wired into the existing `SpecialistRole`
Literal / `SPECIALIST_ROLES` roster in `maf_starter/orchestration.py` and
`team_factory.py`. That roster is a strict, exhaustively-mapped typed pipeline
(`STAGE_AGENT_MAP`, `SPECIALIST_STAGE_MAP`, `SPECIALIST_DEFAULT_HANDOFF_TARGETS`
all require exhaustive per-role entries, and `tests/test_phase3_specialists.py`
asserts exact-tuple equality against `SPECIALIST_ROLES`). Ripping that literal
open to add `"critic"` would force edits across roughly two dozen call sites
for a role that runs concurrently and out-of-band from the sequential
planner/researcher/implementer/reviewer chain -- exactly the kind of unbounded
ripple this phase's 2-plan budget forbids. Instead the critic is built as an
independently callable module and CLI, reusable from any caller (a future
orchestrator hook, a git pre-merge check, or a manually invoked review)
without touching the existing typed roster. This satisfies "reusable role in
autogen/MAF team_factory" as stated in scope: it lives beside `team_factory.py`
in the same package and follows its factory conventions (`build_*` naming,
`Settings`-injected construction, `from __future__ import annotations`,
dataclass-based structured returns) without requiring roster surgery.

Output: `maf_starter/critic.py` (the checks plus the `CriticReport` structured
result), `maf_starter/critic_cli.py` (the runnable entrypoint), and
`tests/test_critic.py` (pytest falsifiers for the dirty/clean/malformed paths).
</objective>

  <name>Task 1: Write maf_starter/critic.py -- pattern-based Resilience First checker</name>
  <files>portfolio/autogen/maf_starter/critic.py, portfolio/autogen/tests/test_critic.py</files>
  <behavior>
    - parse_unified_diff(diff_text) returns a list of DiffFile objects. Given unified
      diff text (git diff format) with two changed files, it must produce exactly two
      DiffFile entries, each with the correct file path and its added lines (lines
      starting with a single "+", excluding the "+++" file header line).
    - check_bare_except(added_lines) returns a list of Finding objects. An added line
      matching "except:" (bare) or "except Exception:" / "except Exception as ..." with
      no re-raise, no logging/logger call, and no typed exception reference within the
      next few added lines of the same hunk must produce one BLOCKING finding with check
      name "bare-except". An added except block that re-raises a typed exception and
      logs first must produce zero findings from this check.
    - check_missing_telemetry(added_lines) returns a list of Finding objects. An added
      try/except block whose except body has no log/logger/telemetry call must produce
      one ADVISORY finding with check name "missing-telemetry" (never BLOCKING -- this
      check alone must never raise blocking_count).
    - check_file_size_limit(file_path, added_line_count, limit=400) returns a list of
      Finding objects. A file with more added lines than limit must produce one ADVISORY
      finding with check name "file-size-limit"; a file at or under the limit produces
      zero findings.
    - run_critic(diff_text, *, llm_pass=False, settings=None) returns a CriticReport.
      It must run all three checks over every parsed DiffFile and aggregate results into
      CriticReport(findings, blocking_count, advisory_count, exit_code), where exit_code
      is 1 when blocking_count > 0 and 0 otherwise. A diff containing one bare-except
      violation must yield exit_code 1 and blocking_count 1. A clean diff (typed
      exception, logged, under the size limit) must yield exit_code 0.
    - run_critic must never raise on malformed, empty, or non-diff input. Passing an
      empty string or arbitrary garbage text must return a CriticReport without raising,
      with a single advisory "critic-error" finding and exit_code 0 -- a parse failure is
      not itself proof of a Resilience First violation and must never silently block an
      unrelated PR.
  </behavior>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| diff text (stdin/file) -> critic.parse_unified_diff | Untrusted input: diff text may originate from any PR, including adversarial or malformed content |
| critic_cli.py --diff path -> filesystem | CLI reads an arbitrary local file path supplied by the caller |
| critic findings -> merge-gate decision (consumed in Plan 29-02) | A false positive or false negative here directly affects whether real PRs merge |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-29-01 | Denial of Service | critic.run_critic against malformed/huge/adversarial diff text | mitigate | Single outer try/except Exception at the run_critic boundary degrades to an advisory critic-error finding with exit_code 0 instead of raising or hanging; parse_unified_diff is pure line-scanning with no recursion or backtracking regex that could pathologically blow up on crafted input |
| T-29-02 | Repudiation / false-positive blocking | check_bare_except and check_missing_telemetry pattern matches | accept, with mitigation | Severity is explicitly split into BLOCKING (bare-except only, the one unambiguous Resilience First violation named in the Operating Contract) versus ADVISORY (missing-telemetry, file-size-limit) so a probabilistic pattern match against a merely-stylistic gap cannot silently block a PR; Plan 29-02's evidence run against a real diff is the concrete falsifier for this threat |
| T-29-03 | Tampering | critic_cli.py --diff path reads an arbitrary local file | accept | This is a local developer tool run against local repo checkouts (per ENVIRONMENT: no Azure, local execution); the caller already has local filesystem access equal to or greater than what this script can read, so no privilege boundary is crossed |
| T-29-04 | Information Disclosure | Findings output (stdout human report) | accept | Findings are file paths, line content, and diff excerpts already visible in the PR diff itself; no secrets are read or synthesized by the critic |
</threat_model>
