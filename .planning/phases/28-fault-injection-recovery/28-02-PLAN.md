<objective>
Give the Microsoft Agent Framework workers structured JSON failure telemetry on
stderr, proven by a fault-injection test that simulates an API outage and captures
the log output. Today, provider_fallback.py's fallback middleware and
worker_boundary.py's background-task runner both catch exceptions correctly (the
try/except boundaries already exist) but neither writes anything to a log stream —
failures are only visible via in-memory response metadata
(additional_properties["route_attempts"]) or an in-memory status dict
(WorkerBoundary._status[run_id] = f"error:{exc}"), which is invisible to any
process-external log-based monitoring. This closes that gap for REQ-1.4.3. This
plan also closes the one remaining C6 gap: loop_worker_cli.py already enforces
MAX_REQUEST_BYTES on stdin, but provider_fallback.py's CLI-fallback subprocess
path (_run_subprocess / _messages_to_prompt) has no size limit on captured stdout
or constructed prompts.

Purpose: REQ-1.4.3 requires "All Microsoft Agent Framework workers must include
try/catch blocks that emit structured JSON telemetry on failure, proven by log
output during a simulated API outage." The try/catch blocks already exist (verified
in provider_fallback.py's fallback_middleware/_wrap_stream_with_fallback and
worker_boundary.py's WorkerBoundary._run); what is missing is the telemetry
EMISSION to an observable log stream. This plan adds one small, shared, well-tested
emission point and wires it into both existing catch blocks — it does not restructure
the fallback/retry logic itself (that logic already works correctly, per the
existing passing tests in test_phase3_routing.py and test_worker_boundary.py).

Output: telemetry.py (new, minimal), three call-site wirings (provider_fallback.py
non-streaming path, provider_fallback.py streaming path, worker_boundary.py),
CLI subprocess/prompt size guards, and new fault-injection tests proving a
simulated outage produces parseable structured JSON on stderr.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| External provider API (Gemini/Anthropic/Ollama/CLI) -> fallback_middleware | Untrusted call outcome (success, quota error, timeout, malformed response) crosses into MAF's chat-middleware control flow |
| CLI subprocess stdout -> _run_subprocess return value | Untrusted, attacker- or bug-influenced subprocess output (gemini-cli/claude-cli/codex-cli) crosses into the response pipeline as plain text |
| Conversation history -> subprocess argv/stdin | Untrusted, unbounded accumulated conversation content crosses into a subprocess invocation via _messages_to_prompt |
| Background workflow exception -> WorkerBoundary status dict | Untrusted failure detail (exception message, potentially containing secrets from upstream error text) crosses into both an in-memory dict and now a stderr log line |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28-05 | Denial of Service | fallback_middleware / _wrap_stream_with_fallback | mitigate | Task 1 proves a simulated API outage produces observable structured telemetry at each stage (primary failure, per-step failure, exhaustion or recovery) instead of a silent in-memory-only failure that operators cannot detect without reading response payloads |
| T-28-06 | Denial of Service | _run_subprocess CLI fallback output capture | mitigate | Task 2 adds MAX_CLI_OUTPUT_BYTES cap so a runaway or malicious CLI subprocess cannot return unbounded output that exhausts memory or downstream log/telemetry storage |
| T-28-07 | Denial of Service | _messages_to_prompt CLI prompt construction | mitigate | Task 2 adds MAX_CLI_PROMPT_BYTES cap so an unbounded conversation history cannot overflow subprocess argv/stdin limits or cause the CLI subprocess to hang |
| T-28-08 | Denial of Service | WorkerBoundary._run background task failures | mitigate | Task 2 adds structured telemetry emission alongside the existing in-memory error: status string, making background-task failures visible to log-based monitoring, not just to callers polling get_status |

| T-28-SC | Tampering | pip installs | accept | No new third-party packages introduced; telemetry.py uses only stdlib (json, sys, datetime) |
