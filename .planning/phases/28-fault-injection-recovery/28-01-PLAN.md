<objective>
Prove — with real, automated tests, not manual inspection — that gsd-orchestrator's
existing typed-failure machinery (WorkflowState.Failed, FailedState, FailureReason,
RetryCount, TerminalStopReason) and its checkpoint atomicity (temp-file + flush +
atomic rename, added in commit 795242f) actually survive three catastrophic fault
classes: (1) an MCP/API call failing mid-goal, (2) a checkpoint file corrupted after
a partial/interrupted write, and (3) a process killed between the temp-file write and
the atomic rename. This is the REQ-1.4.2 fault-injection E2E verification half.

model this phase verifies. Phase 27 has not yet been planned/executed in this

public enum WorkflowState { Idle, Triaging, Analyzing, Branching, Editing,
  TestGenerating, Validating, Committing, PrCreating, Reviewing, Documenting,
  Done, Failed }

public enum TerminalStopReason { None, Ambiguity, DestructiveAction, ExternalAction,
  PolicyDenied, BudgetExhausted, NoProgress, RuntimeExceeded, Unknown }

public sealed record GsdWorkflowContext {
  string SchemaVersion { get; init; } // "1.1", legacy "1.0" auto-upgraded
  WorkflowState? FailedState { get; init; }
  string? FailureReason { get; init; }
  TerminalStopReason StopReason { get; init; }
  // RetryCount, History, WorkflowId, Issue also present — see full file.
}

<!-- GsdStateMachine.cs (already implemented) -->
public sealed class GsdStateMachine {
  private const int MaxRecoveryAttempts = 1;
  public Task<GsdWorkflowContext> RunAsync(string owner, string repo, int issueNumber, bool triageModeOnly, CancellationToken ct);
  public Task<GsdWorkflowContext> ResumeAsync(string workflowId, CancellationToken ct);
  // catch (Exception ex) in ExecuteLoopAsync sets CurrentState=Failed, FailedState=previousState,
  // FailureReason=ex.Message (truncated to 1024 chars), saves checkpoint, then returns —
  // it does NOT rethrow. This IS the typed-failure boundary to test.
  // ResumeAsync throws InvalidOperationException when RetryCount >= MaxRecoveryAttempts —
  // this IS the deterministic-halt boundary to test.
}

<!-- FileCheckpointStore.cs (already implemented, hardened in commit 795242f) -->
public sealed class FileCheckpointStore : ICheckpointStore {
  // SaveAsync: writes to "{path}.tmp" with FileShare.None, JsonSerializer.SerializeAsync,
  // fs.Flush(flushToDisk: true), THEN File.Move(tmp, path, overwrite: true).
  // LoadAsync: JsonSerializer.DeserializeAsync -> throws JsonException on malformed JSON
  // (uncaught in current code — this plan proves what actually happens, does not assume).
  // ValidateSchemaVersion throws InvalidDataException on unsupported (non-1.0/1.1) SchemaVersion.
  public Task SaveAsync(GsdWorkflowContext ctx, CancellationToken ct = default);
  public Task<GsdWorkflowContext?> LoadAsync(string workflowId, CancellationToken ct = default);
}

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP/GitHub API response -> GsdStateMachine | Untrusted external call outcome (success/failure/timeout) crosses into the state machine's control flow via IWorkflowState.ExecuteAsync |
| Local disk (.gsd/state/*.json) -> FileCheckpointStore.LoadAsync | Untrusted bytes-on-disk (could be corrupted by disk failure, killed process, or manual tampering) cross into JSON deserialization and typed context reconstruction |
| Process lifetime -> checkpoint atomicity | An external actor (OS scheduler, user Ctrl+C, crash, OOM-kill) can terminate the process at any byte offset during SaveAsync |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28-01 | Denial of Service | GsdStateMachine.ExecuteLoopAsync | mitigate | Fault-injection test (Task 1) proves an MCP/API exception mid-goal reaches the catch block and produces WorkflowState.Failed rather than an unhandled crash that would kill the whole orchestrator process |
| T-28-02 | Tampering | FileCheckpointStore checkpoint files on disk | mitigate | Task 2 tests prove corrupted/truncated JSON is rejected with a typed JsonException/InvalidDataException rather than silently accepted as a partially-valid context, which would let a tampered or half-written checkpoint resume execution with wrong state |
| T-28-03 | Denial of Service | Repeated resume of a permanently-failing workflow | mitigate | Task 1 Test 2 proves MaxRecoveryAttempts=1 produces a deterministic halt (InvalidOperationException) instead of infinite retry loops that could exhaust API quota or compute |
| T-28-04 | Information Disclosure | FailureReason stored in checkpoint + posted to GitHub issue comment (PostFailureCommentAsync) | accept | Truncation to 1024 chars (Task 1 Test 3) bounds size but does not redact secrets that might appear in an exception message; out of scope for this phase since it requires touching exception-construction sites across the codebase (Phase 27 concern), not the fault-injection harness itself |
| T-28-SC | Tampering | NuGet installs | accept | No new NuGet packages introduced; xUnit/NSubstitute already referenced |
</threat_model>
