## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| This plan's 6 repos -> org-dotgithub's pinned SHA | Each caller workflow trusts a single, specific commit in a repo it does not own |
| release-please-action -> cas-contracts's existing hand-maintained CHANGELOG.md | First automated run must not silently discard or corrupt hand-written "## [Unreleased]" content |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-39-08 | Tampering | Manifest bootstrap value per repo | mitigate | Every bootstrap value in the interfaces table above was verified against GitHub's real tag API response (`gh api .../tags`), not assumed or copied from a possibly-stale local clone — this specifically avoids repeating the mistake a naive bootstrap would make on cas-reference-product-style repos with local-only tags (see 39-01 T-39-06; not applicable to this group, cas-contracts's tag is confirmed real) |
| T-39-09 | Repudiation (wrong notes attribution) | cas-contracts's first release-please run against an existing hand-written CHANGELOG.md | accept | release-please is designed to detect and append to an existing Keep a Changelog-format file; the existing "## [Unreleased]" section entries were themselves written using the same conventions release-please expects, so first-run divergence risk is low. Reviewer verifies the resulting release PR's CHANGELOG diff before merging, same as any other PR |
| T-39-10 | Denial of Service (tag storm) | 6 simultaneous new release-please triggers | accept | Each repo's workflow only fires on its own `push` to its own default branch (post-merge, already review-gated); no cross-repo trigger chaining exists, so 6 repos wiring up in the same wave cannot compound into a single runaway loop |
</threat_model>
