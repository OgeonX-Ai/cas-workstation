    - Case 1 (red fixture): tag `v0.1.0` created on a commit committed 45 days ago (via

      commit added on top today, untagged -> expect a 'release-stale' finding whose Detail
      matches `v0\.1\.0` and contains a day count > 30.
    - Case 2 (no false positive): a repo whose only tag is on the current HEAD commit (zero
      commits since the tag), regardless of tag age -> expect NO 'release-stale' finding
      (nothing has merged since the release, so it cannot be stale by the requirement's own
      definition: ">30 days since last release that has MERGED CHANGES").
    - Case 3 (no false positive): a repo with a tag less than 30 days old and commits since
      it -> expect NO 'release-stale' finding (recency, not merge-count, is the gate).
    - Case 4 (no tag at all): a repo with commits but no SemVer-shaped tag at all -> expect a
      'release-stale' finding (never released is the maximal case of "stale since release" —

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| GitHub push (merge to default branch) -> release-please-action | Untrusted-until-reviewed commit history crosses into an automated tool that opens PRs and, once a release PR is merged, creates tags/releases with elevated `contents: write` token |
| org-dotgithub reusable workflow -> 12 other repos | Cross-repo supply-chain boundary: a single file in one repo becomes a trust dependency for the release automation of the entire portfolio |
| Local git tag/commit metadata -> workspace-health.ps1 findings | Untrusted local repo state (could be manipulated by a compromised local checkout) crosses into the sweep's pass/fail output that gates CI |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-39-01 | Tampering | release-please version-bump computation | mitigate | Use release-please's own official conventional-commit parser (not a hand-rolled regex script); every release PR is human-reviewed before merge per existing branch protection (required_approving_review_count=1) |
| T-39-02 | Denial of Service (tag storm) | release-please-reusable.yml trigger | mitigate | Reusable workflow only fires on `push: branches: [default]` (post-merge, already review-gated by branch protection); release-please's own default behavior accumulates all pending changes into ONE open release PR rather than tagging per-push — no tag is created until that PR is itself explicitly merged by a human |
| T-39-03 | Repudiation | Release PR's own CI status | accept | GITHUB_TOKEN-authored pushes do not trigger `pull_request`-scoped workflows on other files (GitHub's anti-recursion design); no repo currently has required_status_checks configured, so this does not block merge. Accepted because release PRs only ever touch CHANGELOG.md/.release-please-manifest.json — never source — so the absence of a fresh CI run carries low risk; reviewer manually confirms the last source commit's CI was green before approving |
| T-39-04 | Elevation of Privilege | Reusable workflow's `contents: write, pull-requests: write, issues: write` GITHUB_TOKEN scope, applied org-wide | mitigate | Ephemeral, repo-scoped GITHUB_TOKEN only (no long-lived PAT introduced); the elevated token can open/update a release PR but cannot merge or tag without a human approval, since enforce_admins=true and required_approving_review_count=1 remain in force on every repo |
| T-39-05 | Tampering | Cross-repo reusable-workflow supply chain (org-dotgithub becomes single point of trust) | mitigate | Every caller pins to a specific org-dotgithub commit SHA (never `@main`), and the third-party `googleapis/release-please-action` inside it is itself SHA-pinned (per existing Phase 31 org policy); a future compromise of org-dotgithub's `main` branch cannot retroactively affect already-pinned callers |
| T-39-06 | Repudiation (wrong notes attribution) | Bootstrap manifest baseline for zero-tag repos | mitigate | Manifest baseline verified against real `gh api .../tags` output per repo (not assumed from local-only tags — cas-reference-product's local `v0.1` tag was confirmed absent on GitHub and is NOT used as a baseline); gsd-orchestrator and cas-contracts bootstrap from their real, confirmed-on-GitHub latest tags so their first automated release notes cover only genuine post-tag history |
| T-39-07 | Information Disclosure | workspace-health.ps1 release-stale check reads local git tag/commit timestamps only | accept | No network call, no credential use; same trust level as the file's other local-git-only checks (1-5, 9) |
</threat_model>
